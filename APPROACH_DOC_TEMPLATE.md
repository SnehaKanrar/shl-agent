## 1. Design choices

- **Catalog**: scraped shl.com's product catalog, "Individual Test Solutions"
  table only ([N] items after de-duplication). Stored as flat JSON with name,
  URL, remote/adaptive flags, test type letters, description, job levels.
- **Retrieval**: TF-IDF (1–2 gram, English stopwords) over name+description+
  job-levels+test-type text, ranked by cosine similarity, with optional
  hard filters on test_type and remote_testing. Chose TF-IDF over embeddings
  for [zero extra latency / no API dependency / good lexical match with
  catalog vocabulary — pick your real reason].
- **Conversation agent**: single LLM call per turn (Groq, Llama 3.3 70B),
  temperature 0, forced to return strict JSON describing intent (clarify /
  recommend / compare / refuse) and a normalized role/skill/seniority
  profile. Python — not the LLM — does the catalog lookup, so every name/URL
  shown to the user is guaranteed to exist in the scraped catalog.
- **Refine handling**: because the API is stateless, the LLM re-derives the
  full profile from the entire message history each turn and is explicitly
  instructed to merge new constraints with previously known ones rather than
  resetting.
- **Compare handling**: fuzzy name lookup against the catalog, then a
  templated response built from stored descriptions/test types — never
  free-generated by the LLM.
- **Refusal**: off-topic/injection detection is a boolean from the LLM; the
  actual refusal text is a fixed string in code, so it can't be steered.
